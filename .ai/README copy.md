# AI Knowledge Repository

This folder contains **evidence-driven engineering artifacts** generated by the PRP workflow.

## Structure

```
.ai/
├── observe/              # Architecture discoveries
│   ├── [feature]-observe.md
│   └── ...
├── explain/              # Evidence-ledger analyses
│   ├── [feature]-explain.md
│   └── ...
├── decide/               # Execution designs
│   ├── [feature]-decide.md
│   └── ...
├── decision-records/     # Long-lived design decisions
│   ├── DR-001.md
│   ├── DR-002.md
│   └── ...
└── README.md             # This file
```

---

## Artifact Types

### 1. OBSERVE Artifacts (`observe/`)
**Purpose:** Discover what exists in the codebase

**Contains:**
- Architecture diagrams
- Runtime flow maps
- Impact analysis
- Dependency chains
-Unknowns

**Generated by:** `/prp-story-create` Phase 1

**Evaporates:** Recreated on each run (ephemeral knowledge)

---

### 2. EXPLAIN Artifacts (`explain/`)
**Purpose:** Evidence-backed analysis

**Contains:**
- Observations with evidence
- Evidence strength ratings (★★★★★)
- Inferences
- Confidence levels
- Risk assessments

**Generated by:** `/prp-story-create` Phase 2

**Evaporates:** Recreated on each run (ephemeral knowledge)

---

### 3. DECIDE Artifacts (`decide/`)
**Purpose:** Execution plans with alternatives

**Contains:**
- Task breakdown (CREATE/UPDATE/etc)
- Alternative approaches
- Validation strategies
- Rollback plans

**Generated by:** `/prp-story-create` Phase 3

**Evaporates:** Recreated on each run (ephemeral knowledge)

---

### 4. Decision Records (`decision-records/`)
**Purpose:** Long-lived design decisions

**Contains:**
- Decision ID
- Problem statement
- Alternatives considered
- Chosen solution
- Reasoning (evidence-backed)
- Consequences

**Generated by:** Manual or post-execution review

**Persists:** Git-tracked (persistent knowledge)

---

## Evidence Hierarchy

When analyzing the codebase, use this evidence strength scale:

| Evidence Type | Strength | Example |
|---------------|----------|---------|
| **Runtime trace** | ★★★★★ | Actual execution path observed |
| **Integration test** | ★★★★☆ | Passing test suite |
| **AST reference** | ★★★★☆ | Abstract syntax tree analysis |
| **Unit test** | ★★★☆☆ | Isolated test passing |
| **Grep/text search** | ★★☆☆☆ | Pattern matching in source |
| **Comment/doc** | ★☆☆☆☆ | Documentation or comment |

**Higher evidence strength → Higher confidence**

## Workflow Integration

### Step 1: Observe
```bash
/prp-story-create
→ Generates .ai/observe/[feature]-observe.md
```

### Step 2: Explain
```bash
(prp-story-create continues)
→ Generates .ai/explain/[feature]-explain.md
```

### Step 3: Decide
```bash
(prp-story-create finishes)
→ Generates .ai/decide/[feature]-decide.md
```

### Step 4: Build
```bash
/prp-story-execute
→ Executes based on artifacts above
→ Generates execution log
```

### Step 5: Verify
```bash
(prp-story-execute validates)
→ Independent validation phase
→ Records pass/fail status
```

### Step 6: Record
```bash
Manual review
→ Creates .ai/decision-records/DR-[N].md
Permanent record of design decision
```

## Example Decision Record

```markdown
# DR-001: Derive Model Field from STATIC_MODEL_ID

## Problem
Response model field was hardcoded, causing drift from actual model ID.

## Alternatives Considered

### Option A: Keep separate logic ✓
Pros: Easier to test
Cons: Requires manual sync

### Option B: Derive from STATIC_MODEL_ID ✓ CHOSEN
Pros: Single source of truth
Cons: Requires change across 3 files

## Decision
Derive all model field references from `STATIC_MODEL_ID`.

## Reasoning
Evidence EL-032 shows STATIC_MODEL_ID is the authoritative source.
Eliminates duplication and prevents drift.

## Evidence
- EL-032: src/adapters.ts:7 - STATIC_MODEL_ID definition
- EL-033: src/server.ts:42 - Response formatting
- EL-034: tests/ - Passing after change

## Consequences
- **Positive:** Single source of truth
- **Negative:** Requires update to 3 files
- **Risk:** Medium (affects multiple endpoints)

## Status
**IMPLEMENTED** (2026-06-20)
Validated by: Tests passing, runtime verified

---
[Next decision record...]
```


## Anti-Patterns to Avoid

### ❌ Don't build a knowledge graph
Too complex, maintenance burden

### ❌ Don't store everything
Only persistent knowledge deserves long-term storage

### ❌ Don't make phases strictly sequential
Allow backtracking when new evidence appears

### ❌ Don't trust low-evidence claims
Evidence strength ★★☆☆☆ or below needs verification

## Benefits

1. **Reduced hallucinations** — Every claim has evidence
2. **Better decisions** — Alternatives documented
3. **Faster onboarding** — New engineers read decision records
4. **Audit trail** — Understand why code ended up this way
5. **Compound knowledge** — Evidence ledger grows over time
